# 优先级框架 · v0.1.0

> 解决两个问题：
> 1. **该不该做？**（gating）
> 2. **做的话什么时候做？**（sequencing）
>
> PM Agent 给出任何"做 / 不做 / 先做谁"的建议时必须引用本文档对应章节。

---

## 一、Gating Filter（决定该不该做）

任何想进 backlog 的功能必须依次通过 5 道闸门。任一闸门未过 → KILL_NICELY。

### Gate 1: 战略契合度

- 是否对应主战略文档的某个 Phase 目标？
- 是否符合 `PRODUCT_PRINCIPLES.md`？
- 不通过 → KILL_NICELY，理由模板："此功能与原则 P-X 冲突 / 与 Phase-X 目标无关"

### Gate 2: 机会成本

- 假设我们做这个，会让什么本来该做的事情被推迟？
- 推迟带来的代价 vs. 此功能的收益？
- 不通过 → 推迟到"可以先做"的时机

### Gate 3: 可逆性

- 如果做了发现错了，多久能回滚？
- 决策不可逆 + 不确定性高 → KILL（先做实验/调研）
- 决策可逆 + 高潜力 → DO（小步试错）

### Gate 4: 最小可行验证（MVT）成本

- 不需要做完整功能，能否用最便宜的方式（landing page / Wizard of Oz / 手动 / Figma 原型）验证假设？
- 能 → 先做 MVT，再决定是否做完整功能
- 不能 → 估算完整功能成本，再走 Gate 5

### Gate 5: 量化收益估算

- 用 RICE 评分（见下文）
- 分数 < 阈值 → backlog 末尾或 KILL
- 阈值见下文

## 二、RICE 评分

```
RICE = (Reach × Impact × Confidence) / Effort
```

| 维度 | 单位 | 取值 |
|---|---|---|
| Reach | 30 天内受影响的用户数 | 整数 |
| Impact | 单用户影响 | 0.25 / 0.5 / 1 / 2 / 3 |
| Confidence | 我们对估算的信心 | 0.5 / 0.8 / 1.0 |
| Effort | 单 Agent-周 工作量（PM + 1 dev + 1 designer 协作折算） | 整数 |

### 阈值（基于 weWatch 当前阶段动态调整）

| Phase | 进 backlog 阈值 | 进当前 sprint 阈值 |
|---|---|---|
| Phase 0 | RICE ≥ 8 | RICE ≥ 25 |
| Phase 1 | RICE ≥ 12 | RICE ≥ 40 |
| Phase 2 | RICE ≥ 20 | RICE ≥ 60 |
| Phase 3+ | RICE ≥ 30 | RICE ≥ 80 |

阈值随 phase 上调 = 用户基数变大，做"小事"代价升高，必须更聚焦。

### Reach 估算参考

- Phase 0 / 1 早期：日活 < 500
- Phase 2 中期：DAU 1000-3000
- Phase 3：DAU 10000+
- 设计师侧：通常是消费者侧的 5-10%

### Impact 评分指南

| 分值 | 含义 | 例子 |
|---|---|---|
| 3 | 重塑核心循环 | 编辑器上线、AI 生成上线 |
| 2 | 显著提升留存/转化 | 一键多设备适配、订阅会员 |
| 1 | 中等改进 | 增加搜索过滤维度 |
| 0.5 | 小幅改进 | 优化加载动画 |
| 0.25 | 边缘 | 增加冷门字段 |

### Confidence 评分指南

| 分值 | 含义 |
|---|---|
| 1.0 | 有数据 / 用户访谈支持 |
| 0.8 | 有竞品验证 + 强逻辑 |
| 0.5 | 主要靠直觉 |

> Confidence < 0.5 的项目应先转为 MVT / 调研而非直接做。

## 三、紧急分诊（P 优先级）

用于 [type=triage] 输入。**与 RICE 不同**：RICE 是"该不该做+先做谁"，P 是"出了事多急"。

| 优先级 | 定义 | 响应 SLA | 中断当前工作？ |
|---|---|---|---|
| P0 | 平台不可用 / 数据丢失 / 法律风险即时 / 收入归零 | < 30 分钟响应 | 必须 |
| P1 | 主要功能受损 / 关键 KPI 显著下滑 / 中等合规风险 | 当日响应 | 视情况 |
| P2 | 单功能影响 / 单设计师阻塞 | 本 sprint 内 | 否 |
| P3 | 优化 / 长尾 bug / 改善 | 进 backlog 排队 | 否 |

**严格规则**：每月 P0 ≤ 3 次（含），否则说明"P0 通胀"，必须重新校准定义。

## 四、决策矩阵（适用于"两难选择"）

### Build vs. Buy vs. Defer

```
              收益高           收益低
高复杂度    Buy（用 SaaS）   Defer
低复杂度    Build（自研）    Defer 或 Build（顺手）
```

例：
- 支付分账 → 高复杂 + 收益高 → **Buy**（Stripe Connect）
- 表盘可视化编辑器 → 高复杂 + 收益高 → **Build**（核心壁垒）
- 多语言翻译 → 低复杂 + 收益高 → **Build**（next-intl）+ Buy（DeepL API 翻译）

### MoSCoW（用于单个 PRD 内部）

- **M**ust have：核心 → 上线必有
- **S**hould have：重要 → 没有也能上，但很快补
- **C**ould have：锦上添花 → 余力做
- **W**on't have（this time）：明确不做（防止隐性蔓延）

每个 PRD 在 §6 范围边界中按 MoSCoW 列出。

## 五、Kill 一个功能的标准流程

PM Agent 提议 KILL 时，必须用 `templates/feature_kill_decision.md` 模板，含：

1. 功能简述
2. 为何当初提出（是创始人 idea / 用户请求 / 竞品启发）
3. 哪些 Gate 没过
4. 替代方案（如有）
5. 给提出人的"善意 KILL"措辞建议（重要：保护人际关系）

KILL 不是失败——它是聚焦。每月至少 KILL 5-10 个 backlog item 才是健康团队。

## 六、当框架不能给出明确答案时

罕见但会发生。流程：

1. PM Agent 明确说"框架无法给出确定建议"
2. 列出 2-3 个候选方向 + 各自风险
3. 标注"需要创始人价值判断"
4. 等待回复，绝不"代为决定"
