# /log-dev — 记录开发进度到 `docs/DEVELOPMENT_LOG.md`

用法：用户输入 `/log-dev`，可附带简短上下文说明，例如 `/log-dev 刚完成了动画背景的代码生成部分`。

## 执行步骤

1. 读取 `docs/DEVELOPMENT_LOG.md`，了解已有日志结构和最近一条记录。
2. 回顾当前对话上下文，提取本轮开发的关键信息：
   - **当前分支**（从 `git branch --show-current` 获取）
   - **目标计划**：本轮试图完成什么
   - **重要完成项**：实际完成了哪些具体工作
   - **架构变更**：文件增删、模块重组、接口变化等结构性改动（表格形式）
   - **测试结果**：哪些测试通过/失败，模拟器验证结果
   - **下一步计划**：接下来待做的事项
   - **注意事项**：Gotchas、踩坑记录、遗留问题
3. 在 `docs/DEVELOPMENT_LOG.md` **文件顶部**（`# weWatch 开发日志` 标题行之后）插入一条新日志（按时间倒序）。
4. 如果今天已有一条日志，用 `### HH:mm 补充` 段落追加，不要为同一天创建多条独立记录。
5. 新日志以 `## YYYY-MM-DD (周X)` 开头，内容使用中文。

## 日志格式

```markdown
## 2026-05-12 (周一)

**分支**: feat/animated-background

### 目标计划
- [x] 计划项 A
- [ ] 计划项 B

### 重要完成项
- 完成了 X 功能的实现

### 架构变更
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| `path/to/file.ts` | 新增 | 变更说明 |

### 测试结果
- vitest: 全部通过

### 下一步计划
- 待做事项 1

### 注意事项
- 需要注意的坑
```

## 初始化

如果 `docs/DEVELOPMENT_LOG.md` 不存在，先创建它，顶部添加标题 `# weWatch 开发日志`。

## 约束
- 不要编造内容，只记录对话上下文中真实发生的事。
- 如果用户没有提供足够信息，省略相应段落。
- 文件路径使用相对于项目根目录的路径。
- 完成后告知用户日志已更新。
